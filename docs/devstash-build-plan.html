Configure Before You Write Code

Third-Party Tool Setup

Create accounts and gather API keys/IDs for all tools before starting Phase 0. Several are blocking dependencies for environment variable setup.

⚠️
Do this first, before any code All tools marked Required below block other phases if not configured. Having keys ready means you configure once and never revisit.
▲
Vercel
Deployment · Free Hobby / Pro $20/mo · REQUIRED
Zero-config Next.js hosting with edge CDN, preview deployments per PR, automatic HTTPS, and serverless functions. The only deployment target that works seamlessly with Next.js App Router and @vercel/og.
  • Create account at vercel.com
  • Connect GitHub repo via Vercel dashboard
  • Set root directory = . (project root)
  • Add all .env variables in Project Settings → Environment Variables
  • Note: VERCEL_URL auto-injected — use for canonical base URL
🔍
Google Search Console
SEO · Free · REQUIRED
Primary SEO health monitor. Tracks indexing coverage, keyword impressions, Core Web Vitals from real users, and crawl errors. Submit sitemap here on first deploy.
  • Go to search.google.com/search-console
  • Add property: URL prefix → https://devstash.me
  • Verify via HTML meta tag (add to app/layout.tsx)
  • Copy verification token → GOOGLE_SITE_VERIFICATION env var
  • Submit sitemap post-launch: /sitemap.xml
📊
Google Analytics 4
Analytics · Free · REQUIRED
Tracks user behaviour, traffic sources, and conversion events (CV viewed, contact form submitted, GitHub link clicks). Uses Measurement ID injected via next/script with lazyOnload strategy.
  • Create property at analytics.google.com
  • Set up Web data stream → devstash.me
  • Copy Measurement ID → NEXT_PUBLIC_GA_ID
  • Configure custom events in GA4 dashboard post-launch
  • Link to Search Console in GA4 for unified reporting
🔷
GitHub + GitHub Actions
Version Control + CI/CD · Free · REQUIRED
Source control and automated CI pipeline. GitHub Actions runs Lighthouse audits, MDX frontmatter linting, TypeScript type-check, and broken link scans on every PR.
  • Create repo: devstash-web (private recommended initially)
  • Set branch protection on main — require CI to pass
  • Create secrets: VERCEL_TOKEN, LHCI_GITHUB_APP_TOKEN
  • Install Lighthouse CI GitHub App for PR comments
  • Configure Dependabot for automatic dependency updates
🦁
Bing Webmaster Tools
SEO · Free · RECOMMENDED
Covers Bing + Microsoft Edge search (significant traffic). Can auto-import verified property from Google Search Console — takes 2 minutes after GSC is set up.
  • Go to bing.com/webmasters
  • Import from Google Search Console (easiest path)
  • Submit /sitemap.xml
  • Copy verification meta tag → BING_SITE_VERIFICATION
🔬
Ahrefs Webmaster Tools
SEO · Free (limited) · RECOMMENDED
Backlink monitoring, keyword rank tracking, and site audit. The free tier is sufficient to monitor backlink growth and catch on-page SEO issues at scale.
  • Sign up at ahrefs.com/webmaster-tools
  • Add site → verify via DNS TXT or HTML file
  • Run initial site audit after launch
  • Schedule weekly audit in Ahrefs dashboard
👁️
Microsoft Clarity
UX Analytics · Free · OPTIONAL
Session recordings, heatmaps, and scroll depth analysis. Invaluable for understanding mobile UX and CTA placement. Zero performance cost with lazyOnload strategy.
  • Sign up at clarity.microsoft.com
  • Create new project → devstash.me
  • Copy Project ID → NEXT_PUBLIC_CLARITY_ID
  • Add GDPR note in Privacy Policy
📦
npm + pnpm
Package Management · Free · REQUIRED
Use pnpm over npm for faster installs and deterministic lockfiles. Vercel supports pnpm natively. Enables monorepo-ready architecture for future scaling (e.g. splitting blog into its own package).
  • npm install -g pnpm
  • Create .npmrc with shamefully-hoist=true for Next.js compat
  • Set Vercel install command: pnpm install
  • Commit pnpm-lock.yaml to version control
🤖
Lighthouse CI
Performance CI · Free · RECOMMENDED
Automated performance gating in CI. Every PR is scored against the 90+ Lighthouse target. Blocks merges that would regress performance, accessibility, or SEO scores.
  • Install Lighthouse CI GitHub App
  • Get token → LHCI_GITHUB_APP_TOKEN GitHub secret
  • Create lighthouserc.cjs in project root
  • Add workflow: .github/workflows/lighthouse.yml
🌊
Cloudflare (DNS)
DNS · Free · RECOMMENDED
Point devstash.me DNS to Vercel via Cloudflare for faster DNS propagation, DDoS protection, and flexible DNS management. Keep proxy OFF (grey cloud) on the A record for Vercel to handle SSL properly.
  • Transfer or add domain to Cloudflare
  • Add A record pointing to Vercel IP (from Vercel dashboard)
  • Keep proxy status: DNS Only (not proxied)
  • Vercel handles SSL/HTTPS automatically

Code Quality Config
INFRA
4 steps
  • 1
    Configure ESLint with Next.js recommended rules + custom rules Enable: @next/next/no-img-element (force next/image), no-console (warn), @typescript-eslint/no-explicit-any (error). Add eslint-config-next, eslint-plugin-tailwindcss for class ordering.

P1
Core Infrastructure
Tailwind design tokens, font system, global layout, and CI/CD pipeline
Week 1–2 🔴 Critical — blocking
Tailwind Design Token System
FEINFRA
3 steps
  • 1
    Extend tailwind.config.ts with all DevStash brand tokens Map all color tokens from the brand spec to Tailwind: bg-ds-bg (#0B0F19), bg-ds-surface (#111827), text-ds-accent (#3B82F6), text-ds-purple (#8B5CF6), etc. Add fontFamily tokens. Set darkMode: 'class' for controlled theme switching.
  • 2
    Define CSS custom properties in globals.css Define all tokens as CSS variables on :root for use outside Tailwind (e.g. in SVG, inline styles): --color-accent: #3B82F6, --color-bg: #0B0F19, etc. This ensures tokens are accessible in non-Tailwind contexts.
  • 3
    Configure spacing scale and typography scale in Tailwind Extend spacing with named values: section: '5rem', section-lg: '8rem'. Add fontSize entries matching the type scale: display: ['4.5rem', { lineHeight: '1.05', fontWeight: '700' }].
Font Setup (self-hosted)
FE
2 steps
  • 1
    Configure fonts via next/font/google in app/layout.tsx Load DM Sans (weights 300–700) + JetBrains Mono (weights 400, 500, 700) using next/font/google. Assign as CSS variables: --font-sans, --font-mono. Self-hosting means zero GDPR issues and zero layout shift (no FOUT). Add variable classNames to <html> tag.
  • 2
    Apply fonts to Tailwind fontFamily and globals.css base styles Set html, body { font-family: var(--font-sans); } in globals.css. Configure Tailwind fontFamily.sans and fontFamily.mono to use CSS variables. This means all Tailwind font-sans and font-mono classes automatically use the DevStash fonts.
Root Layout & Metadata Defaults
FESEO
3 steps
  • 1
    Create content/metadata/site.config.ts — global SEO defaults Export: siteName, siteUrl, defaultTitle, defaultDescription, defaultOgImage, twitterHandle. This is the single source of truth for all default metadata — never hardcode these values in page files.
  • 2
    Configure root app/layout.tsx metadata export with all required tags Export metadata object with: title (template), description, metadataBase (from NEXT_PUBLIC_SITE_URL), openGraph, twitter, robots, verification (Google + Bing tokens). Add GA4 and Clarity scripts via next/script with strategy="lazyOnload".

All pages call this function instead of manually constructing metadata. Ensures consistency and prevents missing fields.
GitHub Actions CI Pipeline
INFRAAUTO
3 steps
  • 1
    Create .github/workflows/ci.yml — main CI pipeline Triggers on: all PRs and pushes to main. Steps: pnpm install → TypeScript type check (pnpm tsc --noEmit) → ESLint (pnpm lint) → build (pnpm build). Fail fast — abort on first failure to save CI minutes.
  • 2
    Create .github/workflows/lighthouse.yml — performance gating Runs on: push to main and PRs. Uses @lhci/cli. Test URLs: /, /blog, /projects. Assertions: performance ≥ 90, accessibility ≥ 90, SEO ≥ 90. Posts results as PR comment via Lighthouse CI GitHub App.
  • 3
    Create .github/workflows/link-check.yml — weekly broken link scan Scheduled: every Monday 09:00 UTC (cron: '0 9 * * 1'). Scans all internal links for 404s. Opens a GitHub Issue automatically if broken links are found. This prevents link rot from accumulating silently.

UI Primitives — components/ui/
FE
6 components

FESEO
4 components

MobileNav — drawer/sheet component for mobile navigation. Traps focus when open. Closes on route change (via usePathname effect). Use dialog HTML element for accessibility.
Footer — static Server Component. Navigation links, legal links, social links, copyright. address semantic element for contact info.
Breadcrumb — renders both visible breadcrumb trail and JSON-LD BreadcrumbList schema. Takes items[] array. Used on all nested routes (project detail, blog post, category archive).

SEOFE
3 components

Schema Builders — lib/schema/builders.ts
BESEO
6 builder functions
ℹ️
Install schema types pnpm add schema-dts — provides TypeScript types for all Schema.org types (Person, BlogPosting, SoftwareApplication, etc). Type-safe JSON-LD with zero runtime cost.
buildPersonSchema() — for the About page and root layout. @type: Person with name, jobTitle, url, sameAs links.
buildWebSiteSchema() — for the root layout. Includes SearchAction if site search is implemented.
buildBlogPostingSchema(post: BlogPost) — maps all MDX frontmatter fields to BlogPosting schema. Called in each blog post's generateMetadata.
buildProjectSchema(project: Project) — maps project data to SoftwareApplication or CreativeWork depending on project.type.
buildBreadcrumbSchema(items: BreadcrumbItem[]) — takes array of {name, url} and returns BreadcrumbList. Called from the Breadcrumb component.
buildFAQSchema(faqs: FAQItem[]) — for FAQ sections on any page. Renders both visible accordion UI and invisible FAQPage schema.

MDX Pipeline — lib/markdown/
BE
5 steps
  • 1
    Install and configure MDX toolchain pnpm add @next/mdx @mdx-js/loader @mdx-js/react + pnpm add -D gray-matter remark remark-gfm rehype-slug rehype-pretty-code. Configure in next.config.ts to enable .mdx files. rehype-pretty-code for syntax highlighting (uses Shiki — zero client JS).
  • 2
    Build lib/markdown/blog.ts — blog data fetching layer Export: getAllPosts(options?) — reads all .mdx files from content/blogs/, parses frontmatter with gray-matter, filters draft: false, sorts by createdAt desc. getPostBySlug(slug) — returns full post including parsed MDX content. getRelatedPosts(post, limit?) — returns N posts matching same category or tags.
  • 3
    Build lib/markdown/projects.ts — project data fetching layer Export: getAllProjects(), getProjectBySlug(slug), getFeaturedProjects(limit?). Reads from content/projects/*.json. For scalability: the function signature is identical to what a future CMS SDK would export — so swapping the data source is a 1-file change.
  • 4
    Build lib/automation/ utilities slugify(text) — consistent slug generation. readingTime(content) — calculates estimated reading time at 200 wpm. validateFrontmatter(data) — runtime validation of required fields (for use in the linting script). formatDate(date) — consistent date formatting.
  • 5
    Configure generateStaticParams in all dynamic routes Every dynamic page ([slug], [category], [tag]) must export generateStaticParams(). This pre-renders all content pages at build time as static HTML — no server required at runtime, instant loading, perfect for SEO.

Analytics Wiring

4 steps

Built to Scale

Future-Proofing Decisions

CMS Migration Path
The lib/markdown/blog.ts and lib/markdown/projects.ts data layer has the same exported function signatures as any CMS SDK. When you outgrow file-based content, swap the implementation inside those two files — zero changes to any page or component.
Monorepo Ready
The folder structure and pnpm workspace config means converting to a Turborepo monorepo (e.g. apps/web, packages/ui, packages/content) requires only moving folders, not refactoring logic.
i18n Ready
Using Next.js App Router's app/[locale]/ route group pattern means adding internationalisation requires only adding a [locale] segment to the route structure — no page rewrites needed.
API Versioning
All API routes live under app/api/. When you add products (e.g. a developer tool with an API), version from day one: /api/v1/.... The contact form route at /api/contact/route.ts is a template for all future routes.
